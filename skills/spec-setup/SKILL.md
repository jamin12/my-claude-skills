# Spec 문서 체계 세팅

프로젝트의 도메인 맥락·기획 논의·설계 결정이 휘발되지 않도록, `docs/specs/` 디렉토리와 `CLAUDE.md` 안의 관련 섹션들을 한 번에 세팅한다. 이 스킬은 **1회성 스캐폴딩** 이다. 세팅 이후의 모든 유지보수(신규 도메인 추가, spec 업데이트, 작성 원칙 강제)는 프로젝트 CLAUDE.md가 직접 지시하도록 구성한다.

## 적용 시점

- 새 프로젝트에 처음으로 spec 체계를 도입할 때
- 기존 프로젝트에 spec 문서 관행이 없는 상태에서 체계를 도입할 때
- 사용자가 "spec 세팅해줘", "spec-setup" 등을 요청할 때

이미 `docs/specs/` 디렉토리 + CLAUDE.md에 관련 섹션이 존재하는 프로젝트에는 실행하지 않는다.

---

## CRITICAL — 실행 원칙

#### `setup-idempotent`
같은 프로젝트에서 두 번 실행해도 CLAUDE.md 내용이 중복되거나 기존 내용을 덮어쓰지 않는다. 실행 전 다음을 확인한다:

- `docs/specs/` 디렉토리 존재 여부
- `CLAUDE.md` 에 `필수: 도메인 작업 전 spec 확인` 섹션 존재 여부
- `CLAUDE.md` 에 `맥락 노트 (docs/specs) 유지` 섹션 존재 여부

이미 존재하면 사용자에게 알리고 **중복 삽입하지 않는다.** 부분적으로만 존재하면 부족한 섹션만 추가한다.

#### `setup-no-features-list`
`docs/features.md` 같은 기능 체크리스트는 절대 생성하지 않는다. 기능 상태는 코드에서 확인 가능하므로 드리프트 위험이 있는 문서를 만들지 않는다.

#### `setup-preserve-claude-md`
기존 `CLAUDE.md` 내용을 절대 삭제하거나 재배치하지 않는다. 이 스킬은 **추가**만 한다. 삽입 위치는 기존 `코드 작성/수정 전 skill 확인` 류 섹션 직후가 자연스럽지만, 없으면 파일 끝에 덧붙인다.

---

## HIGH — 실행 순서

### 1. 사전 점검
- `docs/specs/` 디렉토리 존재 확인
- `CLAUDE.md` 존재 확인 (없으면 "CLAUDE.md가 없습니다. 먼저 만들까요?" 물어봄)
- 기존 spec 관련 섹션 중복 확인 (`setup-idempotent`)

### 2. 도메인 인터뷰
사용자에게 프로젝트의 주요 도메인 목록을 물어본다. 각 도메인은 spec 파일 1개에 대응된다.

**질문 예시**
> 이 프로젝트의 핵심 도메인을 알려주세요. 각 도메인은 `docs/specs/{name}.md` 파일로 만들어집니다.
>
> - 도메인 이름 (kebab-case)
> - 그 도메인이 담는 개념 한 줄 (예: "인증, JWT, 세션, 역할")

최소 `overview` 는 항상 포함한다 (프로젝트 전반/아키텍처 배경용).

### 3. `docs/specs/` 디렉토리 및 stub 파일 생성
- 각 도메인에 대해 `docs/specs/{domain}.md` 빈 파일 생성
- stub 내용은 H1 제목 한 줄만 (`# {도메인명} 맥락`). 그 이상 쓰지 않는다 — 스킬이 가짜 맥락을 만들면 안 된다.

### 4. `CLAUDE.md` 에 세 섹션 삽입
아래 **삽입 템플릿** 세 개를 순서대로 삽입한다. 도메인 매핑 테이블은 2단계에서 받은 목록으로 채운다.

### 5. 완료 안내
다음을 사용자에게 알린다:

- 추가된 섹션 목록
- 생성된 spec 파일 목록
- "이후 새 도메인이 생기면 CLAUDE.md의 매핑 테이블에 한 줄씩 추가하시면 됩니다. 유지보수는 프로젝트 CLAUDE.md가 알아서 지시합니다" 안내

---

## HIGH — 삽입 템플릿

아래 세 개 섹션은 **있는 그대로 복사해서** `CLAUDE.md` 에 삽입한다. `{...}` 부분만 인터뷰 결과로 채운다. 임의로 문구를 바꾸거나 요약하지 않는다 — 이식 결과가 프로젝트마다 달라지면 "같은 체계" 의미가 없어진다.

### 템플릿 1: 필수 spec 확인 섹션

```markdown
## 필수: 도메인 작업 전 spec 확인

도메인 코드를 작성/수정하기 전에 해당 spec 파일을 읽어 **기획 맥락과 설계 결정**을 파악한다. spec에는 코드에서 볼 수 없는 "왜 이렇게 만들었는지"가 담겨 있다.

| 작업 도메인 | 반드시 읽을 spec |
|-------------|------------------|
| 프로젝트 전반 / 용어 / 아키텍처 배경 | `docs/specs/overview.md` |
| {도메인 1 설명} | `docs/specs/{domain-1}.md` |
| {도메인 2 설명} | `docs/specs/{domain-2}.md` |
| ... | ... |
```

### 템플릿 2: 맥락 노트 유지 섹션

```markdown
## 맥락 노트 (docs/specs) 유지

기능 구현 중 사용자와 나눈 도메인 맥락, 기획 논의, 설계 결정 사항은 `docs/specs/` 의 해당 도메인 spec 파일에 지속적으로 반영한다.

- **기존 도메인**: 해당 spec 파일을 업데이트한다
- **새 도메인**: 새 spec 파일을 생성하고, 위 `필수: 도메인 작업 전 spec 확인` 테이블에 한 줄 추가한다
- **반영 시점**: 기획 논의에서 새로운 규칙, 예외 케이스, 설계 결정이 확정될 때마다 즉시 반영한다
- **목적**: 다음 대화에서도 동일한 맥락을 유지하여 반복 설명 없이 작업을 이어갈 수 있도록 한다
```

### 템플릿 3: 작성 원칙 섹션

```markdown
### docs/specs 작성 원칙

**코드에서 볼 수 있는 내용은 쓰지 않는다.** API 경로, DB 컬럼, 계산 공식, 클래스 구조 등은 코드를 읽으면 알 수 있다. 대신 코드만으로는 알 수 없는 다음 내용을 중심으로 작성한다:

| 써야 하는 것 | 쓰지 않는 것 |
|-------------|-------------|
| 왜 이 기능을 만들게 되었는지 (비즈니스 배경) | API 경로, Method, Request/Response 구조 |
| 검토했던 대안들과 기각 이유 | DB 테이블/컬럼 정의 |
| 채택한 방식의 트레이드오프 | 계산 공식, 로직 설명 |
| 개발 과정에서의 설계 판단과 그 근거 | 클래스/패키지 구조 |
| 남은 과제, 알려진 한계 | 코드에서 바로 볼 수 있는 사실 |

핵심은 **"우리가 왜 이렇게 만들었는지"를 미래의 개발자(또는 AI)가 이해할 수 있게** 쓰는 것이다.
```

**삽입 순서와 위치**
- 템플릿 1은 독립 섹션으로 삽입한다
- 템플릿 2와 3은 연속으로 삽입한다 (템플릿 3은 템플릿 2의 하위 섹션이므로 H3)
- 가능하면 모두 한 블록으로 같이 두어 "spec 관련 지시가 한 곳에 모여있음" 을 유지한다

## MEDIUM — 금지사항

#### `forbid-features-md`
`docs/features.md` 또는 그에 준하는 "기능 체크리스트" 파일을 생성·참조하지 않는다. 코드에서 볼 수 있는 내용은 문서화하지 않는 것이 이 체계의 기본 원칙이다.

#### `forbid-fake-content`
spec stub 파일에 가짜 맥락을 채워넣지 않는다. 도메인 이름이 주어졌다고 해서 "이 도메인은 보통 이런 흐름이 있습니다" 같은 일반론을 쓰면 드리프트의 시작이 된다. stub은 H1 한 줄만 둔다.

#### `forbid-maintenance-in-skill`
이 스킬에 "도메인 추가", "spec 업데이트" 같은 유지보수 기능을 붙이지 않는다. 세팅 이후 모든 유지보수는 프로젝트 CLAUDE.md가 담당한다. 스킬이 유지보수까지 맡으면 스킬과 CLAUDE.md 사이에 규칙 중복이 생긴다.

---

## LOW — 네이밍

#### `naming-domain-kebab`
spec 파일명은 소문자 kebab-case 를 사용한다. `master-data.md`, `daily-work.md`, `user-servicecode.md` 처럼. 파일명이 곧 도메인 식별자가 된다.

```
# WRONG
docs/specs/MasterData.md
docs/specs/daily_work.md
docs/specs/UserServicecode.md

# CORRECT
docs/specs/master-data.md
docs/specs/daily-work.md
docs/specs/user-servicecode.md
```
